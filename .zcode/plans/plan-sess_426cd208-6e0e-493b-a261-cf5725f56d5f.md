## Fix M1: Backfill NULL `thread_id` on migration

**File:** `src/scripts/cache/db.py` — `_migrate()` (lines 135–153)

**Problem:** The migration adds the `thread_id` column but never backfills existing rows. New list views use `PARTITION BY v.thread_id`, and SQL groups all NULLs into a single partition — so upgrading users see their inbox collapse to ~1 message until the next background sync heals it.

**Change:** Add one `UPDATE` inside the existing `_migrate` `with _connect` block, after the ADD COLUMN loop and CREATE INDEX (so `thread_id` is guaranteed to exist), before setting `user_version`:

```python
# Backfill legacy rows: SQL window functions group all NULL thread_ids
# into a single partition, which collapses the list view for users
# upgrading from a schema that didn't persist thread_id.
conn.execute("UPDATE emails SET thread_id = message_id_hash WHERE thread_id IS NULL")
```

This runs only for DBs below `_SCHEMA_VERSION` (7), is idempotent (only touches NULL rows), and is safe because `message_id_hash` is non-null and always populated at insert.

**Why this placement:** The migration already guards on `version >= _SCHEMA_VERSION` returning early (db.py:138–139), so this UPDATE executes exactly once per upgrading DB. No new tests strictly required — the migration is additive and the existing `test_migrate_is_idempotent` covers the path — but I'll add a focused regression test (`test_migrate_backfills_null_thread_id`) to lock the behavior in.
